# 前端接口文档

## 1. 说明

本文档描述当前前端项目已经接入或预留的接口能力，覆盖：

- 本地论文解析接口
- Task1 全局逻辑分析接口
- Task2 规则审计与规则库接口
- 前端本地会话保存逻辑
- 仍处于预留状态的评语/推荐相关接口

当前前端默认通过以下环境变量控制：

| 变量名 | 默认值 | 说明 |
| --- | --- | --- |
| `VITE_API_BASE_URL` | `/api` | 业务后端基础路径 |
| `VITE_PARSER_API_BASE_URL` | `http://127.0.0.1:8000` | 本地论文解析服务基础路径 |
| `VITE_USE_MOCK` | `true` | 是否启用前端 mock 数据 |
| `VITE_USE_LOCAL_PARSER` | `true` | 上传 PDF 时是否优先走本地解析接口 |

当 `VITE_USE_MOCK=false` 时，前端会优先调用本文档中的真实后端接口。

## 2. 本地论文解析接口

### 2.1 上传解析论文

- 方法：`POST`
- 路径：`http://127.0.0.1:8000/papers/parse`
- Content-Type：`multipart/form-data`

#### 请求字段

| 字段名 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `paper` | file | 是 | 论文 PDF 文件 |

#### 响应字段

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| `submissionId` | string | 前端会话提交 ID |
| `paperName` | string | 论文名称 |
| `paperMarkdown` | string | 解析后的 `paper.md` 内容 |
| `paperAssetBase` | string | 图片静态资源基础路径 |
| `paperMeta` | object | 解析后的 `paper_meta.json` 对象 |
| `artifacts` | object | 产物文件路径信息，可选 |

#### 前端使用位置

- 上传页：`src/views/UploadView.vue`

## 3. Task1 全局逻辑分析接口

对应后端文档：
- `documents/Task1_chunked_long_API_RFC_中文版.md`

### 3.1 发起全局逻辑分析

- 方法：`POST`
- 路径：`/api/task1/audit`
- Content-Type：`application/json`

#### 请求体

```json
{
  "mode": "chunked_long",
  "paper_bundle": {
    "paper_md": "# title\n...",
    "paper_meta": {
      "doc_id": "paper_xxx",
      "anchors": []
    },
    "assets": {
      "figures_dir": "assets/figures",
      "tables_dir": "assets/tables"
    }
  }
}
```

#### 前端组装规则

| 字段 | 前端来源 |
| --- | --- |
| `paper_bundle.paper_md` | `submission.paperMarkdown` |
| `paper_bundle.paper_meta` | `submission.paperMeta` |
| `paper_bundle.assets.figures_dir` | 固定传 `assets/figures` |
| `paper_bundle.assets.tables_dir` | 固定传 `assets/tables` |

#### 响应消费方式

前端读取：

```json
data.result.logic_analysis
```

并在页面中拆成两类用途：

1. 工作台第一个流程“全局性问题分析”
   - 使用 `issues[]`
   - 展示问题标题、证据锚点、问题说明、修改建议

2. 工作台第三个流程“修复”中的评审概述
   - 基于 `logic_analysis` 映射为前端的 `reviewSummary`
   - 展示综合结论、亮点、待完善点、后续动作

#### 当前映射说明

由于 Task1 后端文档本身不直接返回“综合评分 / 亮点 / 待完善点 / nextActions”这一套前端展示结构，前端做了轻量映射：

- `issues[].severity` -> 用于推导 `overallScore`
- `core_argument_consistency.conflict_summary` -> 用于概述摘要
- `logic_dimensions` 中 `status=pass` 的项 -> 用于亮点
- `issues[].analysis` -> 用于待完善点
- `issues[].logical_node` / `dimension_keys` -> 映射为修改建议文案

这部分属于前端展示层加工，不改变后端原始返回。

#### 前端使用位置

- 工作台页：`src/views/ReviewWorkspaceView.vue`
- 接口层：`src/services/api.js`

## 4. Task2 规则审计与规则库接口

对应后端文档：
- `documents/task2接口文档.md`

### 4.1 规则审计

- 方法：`POST`
- 路径：`/api/task2/audit/`
- Content-Type：`application/json`

#### 请求体

```json
{
  "paper_text": "# title\n...",
  "meta_data": {
    "doc_id": "paper_xxx",
    "anchors": []
  }
}
```

#### 前端组装规则

| 字段 | 前端来源 |
| --- | --- |
| `paper_text` | `submission.paperMarkdown` |
| `meta_data` | `submission.paperMeta` |

#### 响应消费方式

前端读取：

```json
{
  "msg": "审计成功",
  "report": []
}
```

并在工作台第二个流程“基于规则的问题分析”里做分组展示：

| 标签页 | 分组规则 |
| --- | --- |
| 标准规则库 | `rule_id` 以 `R-SYS` 开头 |
| 自建规则库 | 非 `R-SYS` 且非 `R-AUTO` 的项 |
| 大模型推荐 | `rule_id` 以 `R-AUTO` 开头 |

其中“大模型推荐”标签页中的每条结果，前端支持点击“添加到用户自建库”。

#### 前端使用位置

- 工作台页：`src/views/ReviewWorkspaceView.vue`
- 接口层：`src/services/api.js`

---

### 4.2 获取规则库列表

- 方法：`GET`
- 路径：`/api/task2/get_rules/`
- Query：`status=approved|pending|rejected`

#### 当前前端使用方式

当前规则库管理页默认调用：

```http
GET /api/task2/get_rules/?status=approved
```

并将后端返回的规则对象映射为前端规则卡片结构：

| 前端字段 | 后端字段 |
| --- | --- |
| `id` | `rule_id` |
| `title` | `rule_name` |
| `category` | `dimension` |
| `summary` | `description` 或 `prompt_fragment` |
| `tags` | `triggers` |
| `defaultSelected` | `is_active` |

#### 前端使用位置

- 规则库管理页：`src/views/RuleLibraryManagementView.vue`
- 工作台页首次进入时的规则库兜底加载：`src/views/ReviewWorkspaceView.vue`
- 接口层：`src/services/api.js`

---

### 4.3 从评审文本中抽取规则

- 方法：`POST`
- 路径：`/api/task2/extract/`
- Content-Type：`application/json`

#### 请求体

```json
{
  "review_text": "专家评审意见文本"
}
```

#### 当前前端组装规则

前端允许用户：

- 上传 `txt/md` 文件
- 或直接在文本框中粘贴内容

最终统一抽取出纯文本，传给后端：

```json
{
  "review_text": "..."
}
```

#### 响应消费方式

后端返回：

```json
{
  "msg": "规则抽取完成",
  "extracted_total": 1,
  "saved_to_pending": 1,
  "rules": []
}
```

前端会把 `rules[].rule_name` 优先提取出来，拼成多行文本后展示到“抽取后的规则编辑框”中，等待用户确认提交到本地自建规则列表。

#### 前端使用位置

- 规则库管理页：`src/views/RuleLibraryManagementView.vue`
- 接口层：`src/services/api.js`

---

- 方法：`POST`
- 路径：`/api/task2/toggle_rule/`
- Content-Type：`application/json`

#### 请求体

#### 前端使用位置

- 接口层：`src/services/api.js`

## 5. 前端本地保存逻辑

当前有一部分配置并没有对应的后端接口，前端先保存在会话层：

| 功能 | 当前实现 |
| --- | --- |
| 已选标准规则 | `sessionStorage` |
| 用户自建规则列表 | `sessionStorage` |
| 规则编辑态草稿 | 页面局部状态 |
| 最近保存时间 | `sessionStorage` |

相关状态文件：

- `src/stores/reviewSession.js`

说明：

- 当前 `saveRuleSelection()` 已调整为前端本地保存行为，不再请求不存在的后端接口
- 页面顶部“保存当前规则配置”本质上是前端确认保存到当前会话

如果后端后续新增“保存规则选择结果”的正式接口，可以在此基础上再补提交逻辑。

## 8. 联调建议

建议联调顺序：

1. 启动本地解析服务，确认上传页能返回 `paperMarkdown + paperMeta`
2. 打开 `VITE_USE_MOCK=false`
3. 验证 Task1：
   - 第二页第一个流程能拿到 `issues[]`
   - 第三个流程左侧评审概述能正常显示

   - 规则库管理页能获取 `approved` 规则列表

   - 第二页第二个流程能拿到 `report[]`
5. 最后再接推荐论文相关接口
