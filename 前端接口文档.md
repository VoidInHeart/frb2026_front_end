# 前端接口文档（v2 运行态）

版本：v2.0  
更新日期：2026-04-13  
状态：联调基线（替代旧版 task1/task2 分散接口说明）

---

## 1. 文档目标

本文档定义当前前后端联调使用的统一运行态接口。

核心流程：

1. 前端提交论文，后端立即返回受理成功信号和 run_id。
2. 前端按四个阶段分别请求阶段结果：格式、逻辑、创新、汇总。
3. 前端可随时查询运行进度与状态。
4. 汇总阶段的优秀论文推荐接口先占位，后端暂不实现推荐生成。

说明：

1. 本文档优先描述业务后端接口，不包含 PDF 解析服务实现细节。
2. 若本地解析仍独立部署，可继续保留 /papers/parse，不影响本协议。

---

## 2. 基础约定

### 2.1 Base URL

1. 业务后端基础路径：/api
2. 示例本地地址：http://127.0.0.1:8000/api

### 2.2 通用响应包装

建议统一返回结构：

```json
{
  "ok": true,
  "code": "OK",
  "message": "success",
  "data": {}
}
```

错误时示例：

```json
{
  "ok": false,
  "code": "RUN_NOT_FOUND",
  "message": "run not found",
  "data": null
}
```

### 2.3 阶段命名（前端）

1. format
2. logic
3. innovation
4. summary

### 2.4 阶段映射（后端内部）

1. format -> task2_prefilter
2. logic -> task1
3. innovation -> task2_refine
4. summary -> recommendation_e5

### 2.5 空值与降级约定（必须遵守）

在“重大问题提前汇总”或“阶段被跳过”场景，部分阶段会没有有效分析内容。为避免前端崩溃，约定如下：

1. run 只要能产出汇总结果，最终状态仍为 completed。
2. 阶段被跳过时，stage_status 使用 skipped，不使用 failed。
3. 容器字段优先返回空容器，不返回 null：
   1. 数组字段返回 []。
   2. 对象字段返回 {}。
4. 必备字段不得缺省：
   1. result_summary 必须存在。
   2. result_summary.merged_issues 至少返回 []。
   3. result_summary.warnings 至少返回 []。
5. recommendation 未实现时必须返回占位结构 implemented=false + items=[]。
6. 兼容旧数据：若历史响应出现 null，前端按空对象/空数组兜底转换，不抛异常。

阶段空结果建议结构：

```json
{
  "stage_status": "skipped",
  "stage_output": {
    "available": false,
    "reason_code": "STAGE_SKIPPED",
    "warnings": []
  }
}
```

## 3. 运行生命周期接口

## 3.1 创建运行（提交论文并受理）

方法：POST  
路径：/api/runs

请求体示例：

```json
{
  "paper_title": "Paper Title",
  "paper_bundle": {
    "paper_md": "# Title\\n...",
    "paper_meta": {
      "doc_id": "paper_xxx",
      "paper_type": "research",
      "anchors": []
    },
    "assets": {
      "figures_dir": "assets/figures",
      "tables_dir": "assets/tables"
    }
  },
  "runtime_context": {
    "capability_config": {
      "recommendation": {"mode": "off"}
    }
  }
}
```

响应体示例：

```json
{
  "ok": true,
  "code": "RUN_ACCEPTED",
  "message": "run accepted",
  "data": {
    "run_id": "7f3d2c9e-...",
    "status": "created",
    "current_stage": "format",
    "accepted_at": "2026-04-13T10:00:00Z"
  }
}
```

阶段被跳过时示例：

```json
{
  "ok": true,
  "code": "OK",
  "message": "success",
  "data": {
    "run_id": "7f3d2c9e-...",
    "stage_name": "innovation",
    "stage_status": "skipped",
    "stage_output": {
      "available": false,
      "reason_code": "STAGE_SKIPPED",
      "warnings": []
    }
  }
}
```

说明：

1. 该接口只负责受理与分配 run_id，不要求等待全流程完成。
2. 若后端有预解析过程，可返回 status=parsing；前端通过状态接口轮询。

## 3.2 查询运行状态（进度/当前阶段）

方法：GET  
路径：/api/runs/{run_id}/state

响应体示例：

```json
{
  "ok": true,
  "code": "OK",
  "message": "success",
  "data": {
    "run_id": "7f3d2c9e-...",
    "status": "running",
    "current_stage": "logic",
    "progress": 50,
    "stage_runs": [
      {"stage_name": "format", "status": "completed"},
      {"stage_name": "logic", "status": "running"},
      {"stage_name": "innovation", "status": "pending"},
      {"stage_name": "summary", "status": "pending"}
    ],
    "last_error": ""
  }
}
```

状态建议值：

1. created
2. parsing
3. running
4. waiting
5. completed
6. failed
7. aborted

运行态补充字段建议：

1. next_stage：当前允许进入的下一阶段（format/logic/innovation/summary）
2. allowed_actions：当前运行态允许动作（continue/skip/abort/retry）
3. block_reason_code：当不可继续时返回（如 STAGE_NOT_READY、RUN_ABORTED）
4. block_reason_message：用于前端提示文案（展示用）

建议补充响应示例：

```json
{
  "ok": true,
  "code": "OK",
  "message": "success",
  "data": {
    "run_id": "7f3d2c9e-...",
    "status": "waiting",
    "current_stage": "logic",
    "next_stage": "logic",
    "allowed_actions": ["continue", "skip", "abort"],
    "block_reason_code": "STAGE_NOT_READY",
    "block_reason_message": "当前阶段尚未放行，请先执行人工决策。"
  }
}
```

## 4. 四阶段接口

## 4.1 触发并获取阶段结果

方法：POST  
路径：/api/runs/{run_id}/stages/{stage_name}

路径参数 stage_name 仅允许：

1. format
2. logic
3. innovation
4. summary

请求体示例：

```json
{
  "action": "continue",
  "operator": "frontend_user",
  "reason": "manual confirm"
}
```

说明：

1. action 可选，默认 continue。
2. action 可取：continue、skip、abort。
3. 接口应幂等：已完成阶段重复调用时返回最新快照，不重复执行重任务。
4. 若请求的阶段不是 `next_stage`，后端应拒绝执行并返回 `409 + STAGE_NOT_READY`。
5. 若 run 已 `aborted/failed/completed`，后端应返回 `409 + RUN_STATE_CONFLICT`（或等价错误码）。

响应体示例：

```json
{
  "ok": true,
  "code": "OK",
  "message": "success",
  "data": {
    "run_id": "7f3d2c9e-...",
    "stage_name": "logic",
    "stage_status": "completed",
    "execution_mode": "checkpoint_resume",
    "control_state": {
      "status": "completed",
      "pending_stage": "innovation",
      "allowed_actions": ["continue", "skip", "abort"]
    },
    "result": {}
  }
}
```

阶段未就绪错误示例：

```json
{
  "ok": false,
  "code": "STAGE_NOT_READY",
  "message": "requested stage is not allowed now",
  "data": {
    "run_id": "7f3d2c9e-...",
    "requested_stage": "innovation",
    "next_stage": "logic",
    "allowed_actions": ["continue", "skip", "abort"]
  }
}
```

## 8. 前端联调建议

1. 创建运行后立即保存 run_id 到会话状态。
2. 每次进入阶段页先 GET state，再决定是否 POST 当前阶段执行。
3. 阶段执行中建议 1-2 秒轮询一次 state。
4. summary 页面先展示 result_summary，再请求 recommendations 占位接口。
5. 接口失败时按 code 做分级提示，不直接依赖 message 文案。
6. 前端按钮启用规则：仅当 `next_stage===当前阶段` 且 `allowed_actions` 包含 `continue` 时，才允许“进入下一阶段”。
7. 若状态为 `waiting`，页面应展示“继续/跳过/终止”三按钮，不允许直接跳到后续阶段。
8. 若状态为 `aborted/failed`，禁止所有后续阶段触发，仅显示“重新创建 run”或“重试”入口。
9. 前端读取阶段结果时必须做空值归一化：null -> {}，null 数组字段 -> []。
10. 若 stage_status=skipped，页面显示“该阶段被策略跳过”，而不是“执行失败”。

---

## 8.1 阶段状态机（前后端统一）

最小状态机约定：

1. created/parsing -> format
2. format completed -> logic（当升级条件满足）
3. logic completed -> innovation（当升级条件满足）
4. innovation completed -> summary
5. summary completed -> run completed
6. 任意阶段可进入 waiting（需人工决策）
7. 任意阶段可进入 aborted/failed（流程终止）

约束：

1. 前端不得直接跳阶段触发执行。
2. 后端必须作为最后防线校验阶段顺序，不满足时返回 `STAGE_NOT_READY`。
3. 前端收到 `STAGE_NOT_READY` 后必须回退到 state 同步，不应重试同请求。

## 9. Django 落地建议（最小改造）

1. 保留 Django 作为 HTTP 层，不改现有核心算法服务。
2. 新增一个 reviewapi app，仅做请求适配与响应包装。
3. 在 view 中调用服务层：
   1. ReviewService（单次审查）
   2. StagedReviewService（run/stage/state）
4. 路由建议：
   1. /api/runs
   2. /api/runs/{run_id}/state
   3. /api/runs/{run_id}/stages/{stage_name}
   4. /api/runs/{run_id}/summary
   5. /api/runs/{run_id}/recommendations
   6. /api/runs/{run_id}/decisions
5. 保留旧接口一段时间做兼容转发，待前端全量切换后下线。

## 10. 错误码建议

1. RUN_NOT_FOUND
2. RUN_INVALID_PAYLOAD
3. STAGE_INVALID
4. STAGE_NOT_READY
5. STAGE_EXECUTION_FAILED
6. DECISION_INVALID
7. NOT_IMPLEMENTED
8. INTERNAL_ERROR
9. RUN_STATE_CONFLICT
10. RUN_ABORTED
11. RUN_FAILED

HTTP 状态码建议：

1. 400：请求参数错误（RUN_INVALID_PAYLOAD、STAGE_INVALID）
2. 404：资源不存在（RUN_NOT_FOUND）
3. 409：状态冲突（STAGE_NOT_READY、RUN_STATE_CONFLICT、RUN_ABORTED、RUN_FAILED）
4. 500：后端执行错误（STAGE_EXECUTION_FAILED、INTERNAL_ERROR）
